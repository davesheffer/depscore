What you see before you trust it

The whole supply chain, on one card.

One read-only command. No wrapper around your installs, no account, no telemetry. Just the signals that actually decide whether a package is safe to add.

🛡

Real CVEs, whole tree

Every resolved package matched against npm's own advisory database — the same data npm audit uses. No Snyk account, no AI that hallucinates a vuln.

⚠️

The literal install script

Not "this package has an install script" — the exact command that runs on your machine, flagged in red if it hits the network, pipes to a shell, or evals.

📦

npm-accurate count

A zero-dependency semver resolver (parity-tested against node-semver) resolves every range the way npm does, so the package count matches your lockfile.

🗂️

Audit your whole project

Run it with no argument in any repo to grade your entire dependencies + devDependencies tree at once.

👤

Trust & staleness signals

How many maintainers you'd be trusting, how many deps are deprecated, and when the oldest one last shipped.

🎯

A grade you can gate on

A clear A–F verdict. A critical CVE or a malicious install script forces an F and exit code 2 — drop it straight into CI.

Usage

Nothing to learn.

No flags, no config. Run it, read the card, move on.

# audit one package before you add it
npx depscore express
npx depscore left-pad@1.3.0 # pin a version or range
npx depscore express@^5 # ranges resolved like npm
# audit your whole project's dependency tree
npx depscore
# gate CI — a critical CVE or malicious script exits 2
npx depscore "$NEW_DEP" || exit 1
